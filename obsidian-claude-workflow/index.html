Notes that work back

> how I run my work with Obsidian + Claude Code

A plain-text second brain, with an AI that lives in it

My work notes are an Obsidian vault — just markdown files in folders. Claude Code runs directly on that folder, reading and writing the same files I do.

Tickets/
Obsidian (the memory).
One folder per project. Every project has a map-of-content page, numbered action files with status and due-date metadata, and its meeting notes. Daily notes tie it together.
claude
Claude Code (the assistant).
An AI agent that works on files, not chat history. Pointed at the vault, it can read any note, update any action, draft any document — with full context of everything that came before.
CLAUDE.md
The rulebook.
A standing instructions file teaches Claude my conventions: how action files are structured, how I write to different audiences, even brand standards for documents. Fifteen custom skills package repeatable workflows so one phrase triggers a whole process.

Day-to-day: what it actually does for me

Morning admin
> Here's my diary — set up today.
Claude reads the calendar screenshot, creates the daily note, creates a prepped note for every meeting, links each to prior calls of the same series, and drafts an agenda from what's in the vault.
Action tracking
> Zurich came back — they're holding until Q4. Update the action.
Claude finds the right action file, appends the update, rewrites the status summary, adjusts the due date. 96 actions stay current without me maintaining a tracker.
Comms in my voice
> Draft the weekly update for eSIM.
Claude pulls the week's progress from action files and call notes, and drafts to the house rules it's been taught — short, punchy, non-technical, max three bullets. Every edit I make feeds back into the rulebook, so drafts keep converging on what I'd actually send.
Structured product thinking
> We want to add screen protection to tickets — walk me through this properly.
A skill runs the vague ask through a full product workflow — intake, framing, goals, personas, JTBD, prioritisation, solution definition, roadmap — pausing for my input at each stage. Ends with a branded stakeholder deck, published to a link I can send round.

I supply judgement and decisions; Claude supplies memory, structure, and first drafts. Everything it produces lands back in my system, in my format, ready for the next conversation to build on.

From vague ask to roadmap: the initiative process

One sentence from a stakeholder can trigger a full product discovery process — eleven stages, based on the Product Institute's methodology, each pausing for my judgement before moving on.

> We want to add screen protection to tickets — walk me through this properly.
01 · INTAKE
Understand the ask
Claude grills me for the tacit context a one-line ask never carries — why now, what success looks like, sensitivities. Framed against strategy, given a measurable goal.
02 · DISCOVERY
Understand the people
Stakeholder mapping, then personas and empathy maps for affected customers, then Jobs-to-be-Done to diagnose the real problem — asks are often a solution in disguise.
03 · VALIDATION
Check it's worth it
Scored against customer signal and business alignment, then the riskiest assumptions get lightweight tests — before anything gets built.
04 · DEFINITION
Define and commit
Solution exploration, a story map, the minimum feature set, and a roadmap entry with outcomes and priority.
FINALE
Stakeholder deck
Generated from the artifacts, published to a link I can send round.

Every stage saves its artifact into the vault — the Screen Protection initiative alone produced 15: strategy canvas, personas, job stories, prioritisation, story map, roadmap. Personas reuse across initiatives, so the process compounds.

Discovery discipline I'd struggle to sustain solo, applied in hours instead of weeks — and none of it lives in my head or a chat window. It's all on file for the next initiative to build on.
